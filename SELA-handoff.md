## 二、跨專案通用坑（建議進 Kit）

### 1. 【#75 補強】CDN PWA 的第二種失效模式 + opaque response 陷阱

> **這不是新坑，是 #75 的兩個未涵蓋面 + 第二個證據點（N=2）。**

- **N=2 證據**：RT-Tool（放腫臨床 PWA、醫院網路）獨立命中 #75 的根因，與 RTO-QCC 同 stack 同場景。#75 可升格為「2 個專案踩過」。

- **未涵蓋面 A — 不是 install 失敗，是「永遠不快取」**：
  - #75 描述的症狀是 `addAll` 整個 reject → SW 裝不起來。
  - **RT-Tool 的變體**：App Shell 全是本地檔，`addAll` 從不失敗、SW 裝得起來 —— 但 fetch handler 對跨域直接 `return`（原意是「讓 CDN 走網路」），導致 **CDN 永遠不進 cache**。離線 / 封鎖時 app 開得起來但**完全沒有樣式**，README 宣稱的「離線可用」實為假。
  - **這個變體更隱蔽**：SW 註冊成功、Lighthouse PWA 檢查會過、開發者網路正常時完全看不出來。
  - 做法：fetch handler 要**認得 CDN 網域並納入快取**，不能只做 `origin !== self.location.origin → return`。

- **未涵蓋面 B — opaque response 靜默永不快取**：
  - 用 `mode:'no-cors'` 快取 CDN 時，回的是 **opaque response**：`status === 0`、`ok === false`。
  - 常見寫法 `if (res && res.status === 200) cache.put(...)` → **永遠為 false、靜默不快取**，且沒有任何錯誤訊息。
  - 做法：判斷式必須收 opaque：`if (res && (res.status === 200 || res.type === 'opaque'))`。
  - **這是「靜默失敗」家族**：程式看起來完全正確、跑起來沒報錯、但功能完全沒生效。

- **證據**：RT-Tool V3.5.0 引入 PWA 時寫下、V3.6.2 對齊 Kit 讀到 #75 才發現。
- **檢查 1 結果**：grep Kit「opaque」→ 0 命中（全新）；「max-height」→ 0 命中

### 2. 【新坑候選，N=1】改共用元件的顯隱機制，要 grep 所有直接操作該 class 的地方

- **症狀**：分期頁卡片只顯示標題和半截按鈕、下方內容被裁掉；**工具頁完全正常**（同一個元件、不同頁）。
- **原因（兩層，都要修才會好）**：
  1. 卡片展開動畫用 `max-height: 1400px` 過渡 → 內容超過上限就被 `overflow:hidden` **裁掉**（工具頁內容矮所以沒事、分期頁高就中）
  2. 加動畫時只改了 `toggleCard()`（統一入口），但 **staging.js 有自己的 `remove('hidden')` 直接操作** —— 沒跟著改，與新 CSS 不相容
- **做法**：
  - 摺疊面板**不要用 max-height 做動畫**（必然要猜一個上限、猜錯就裁切）。用 `display:none` ↔ `.open{display:block}` + opacity/transform fade —— 永不裁切、也不必猜高度。
  - 改共用元件的顯隱機制（CSS class）時，**grep 所有直接操作該 class 的地方**一起改，不能只改統一入口函式。
- **可推廣性**：任何有共用摺疊元件 + 多處直接操作 DOM 的前端專案（Vanilla JS 尤其容易，因為沒有型別系統擋）。
- **跟既有坑的關係**：#74 是「引用了不存在的定義」（差集掃描抓得到）；**這條是「定義存在、但繞過統一入口自己來」（差集掃描抓不到）**，兩者互補。
- **N=1**：依 Kit「N=1 暫緩、N≥2 進規範」原則，建議標 `[N=1]` 收在坑庫、或先觀察第二案。
- **檢查 1 結果**：grep Kit「max-height」→ 0 命中

## 三、設計模式 / Claude 失效模式（建議進 philosophy 五.5）

### 1. Claude 憑記憶「修正」既有的領域資料 —— 把對的改成錯的

> **建議收進 `sela-philosophy.md` 五.5「Claude 的已知失效模式」第四條。**

- **本案活例**：查核 OAR 劑量限制時，我「記得」TG-101 大血管 3fx 是 39 Gy，專案裡寫 45 Gy —— 我幾乎要把它「修正」掉。查了權威整合表後確認：**45 Gy 才是對的、我的記憶是錯的**。如果沒查表就改，等於**把一個正確的臨床數值改成錯的**，而且會理直氣壯地在交付訊息裡寫「修正了大血管 3fx」。
- **為什麼這是結構性失效、不是偶發**：
  - Claude 對領域數值有「模糊但自信」的記憶（訓練資料裡看過很多版本、混在一起）
  - 「查核既有資料」這個任務本身會誘發「找出錯誤」的動機 —— **找不到錯會覺得沒做事**，於是把記憶落差當成錯誤
  - 靜默：改完沒有任何機制會發現改錯了（不像程式會報錯）
- **結構性對策（不是語氣加強）**：
  1. **資料類「修正」必須先取得權威來源**（查表 / 官方文件 / 原始論文），**沒有來源就不改** —— 把「我記得是 X」當成**待驗證假設**，不是事實
  2. 交付訊息**明列每條修正的依據來源**（沒來源的不准列進「已修正」）
  3. **「查核後結論是不用改」是合法且有價值的結果** —— 要明寫「對照 X 表，Y 值確認正確、維持原值」，避免下次 Claude 又想改
- **為什麼值得進 Kit**：SELA 的專案多是**醫療 / 臨床工具**（RT-Tool、RTO-QCC、Cancer Navigation、健保藥物、CCM…），這類專案裡「把對的數值改成錯的」是**病人安全等級**的失效，比程式 bug 嚴重。
- **跟既有的差別**：坑庫已有的「憑記憶寫 `old_str` 容易猜錯格式」是**改檔操作層**（改不到、會發現）；這條是**領域資料語意層**（改成功了、但改錯了、不會發現）。
- **檢查 1 結果**：grep Kit「憑記憶」→ 只命中 str_replace 格式那條，語意層無

## 四、Kit 瘦身或調整建議

### 坑庫查重成本：建議速查索引加「症狀關鍵字」欄

- **現狀**：速查索引按「技術棧 / 情境」分類。要判斷「我這條是否已在 Kit」時，實際上是用**症狀關鍵字**去 grep（max-height / opaque / 憑記憶 / localStorage 更名…）。
- **建議**：速查索引每條加一組**症狀關鍵字**（3-5 個 grep 得到的詞）。讓「查重」這個高頻動作從「讀完整份坑庫」降成「grep 索引」。
- **理由**：Kit 已有 75 條坑、還在長。**查重成本會隨坑數線性上升**，而查重是每份 handoff 都要做的動作（Kit 自己也在做，V1.24.0/V1.25.0 的 changelog 都花大篇幅講查重結果）。
- **成本**：一次性補 75 條的關鍵字；之後每條新坑多寫一行。
